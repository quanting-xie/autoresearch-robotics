Experiment 3 — visual analysis (dense reward shaping in replay buffer, success_rate=0.00)

Frames: ./renders/ep{0,1,2}_{start,mid,end}.png

Behavior:
- ep0: gripper starts above the goal (red dot near table edge below it). Mid: arm has retracted to the LEFT and DOWN — gripper poking into the side/back of the base. End: same retracted pose, goal floating to the right unreached.
- ep1: similar to ep0; gripper ends collapsed near the table edge with goal sitting on the table surface ahead.
- ep2: gripper poking down at the table center, goal floating slightly to its right and up.

Key observation: behavior is INDISTINGUISHABLE from exp1 baseline. No goal tracking. Same "arm collapses or retracts to a default pose" failure mode. Dense reward did not change the policy in any visible way.

Comparison vs exp1 (renders_best — same since both 0%):
- mean_distance 0.41 (exp3) vs 0.61 (exp1) vs 0.40 (exp2). Slight improvement but in the noise band of "where untrained policy happens to land." Not goal-tracking.
- total_updates 838 (exp3) vs 853 (exp1) vs 890 (exp2). Same regime.

Why dense reward didn't help — root-cause hypothesis:
- Within 838 gradient updates, auto-alpha is still in early-init regime. INIT_ALPHA's actual init is log_alpha=0 -> alpha=1.0. SAC's actor_loss = (alpha * log_probs - Q).mean(). With alpha=1.0 and FetchReach Q-values around -1 to -50, entropy regularization is COMPARABLE to or DOMINATES the Q-term in early training. The actor is being pushed to maximize entropy (random-ish actions) more than to maximize Q.
- Dense vs sparse reward changes the *quality* of the Q-signal but not the *quantity* of gradient steps. With ~840 updates the Q-network hasn't learned much regardless, AND alpha hasn't annealed down to let the (now-cleaner) Q-signal dominate the actor loss.
- Conclusion: the binding bottleneck is NUMBER OF GRADIENT UPDATES, not signal quality. Need to attack throughput directly.

Highest-leverage attack:
- HIDDEN_DIM 256 -> 128 and N_LAYERS 3 -> 2. Reduces actor+critic from ~410k -> ~30k params (~13x smaller).
- BATCH_SIZE 256 -> 128. Halves per-update FLOPs.
- Combined expected speedup: 3-5x grad steps per second within the 10s budget. Goes from ~850 updates to ~2500-4000 updates. THIS is a different training regime.
- Risk: smaller net might underfit. But for a 13-dim obs / 4-dim action env, 128x2 is still well-specced.

Decision: DISCARD exp 3 (success_rate equal to baseline at 0%). Revert train.py. Keep dense reward as a "promising but insufficient alone" item. Exp 4 attacks update throughput directly via smaller net.

Plan exp 4:
- HIDDEN_DIM = 128
- N_LAYERS = 2
- BATCH_SIZE = 128
- Everything else baseline. Sparse reward retained for clean attribution of the architectural change.
- Predict: total_updates 850 -> 2500+. If this alone breaks 0%, the bottleneck is confirmed as throughput. If still 0%, exp 5 layers dense reward + UTD>1 on this smaller-net base.
